# v2 Deployment Runbook

This runbook is the source of truth for deploying the v2 site from a clean
checkout. Follow it top-to-bottom for first-time setup, then refer to the
later sections for ongoing maintenance.

## Prerequisites

- **Vercel account** with the CLI installed (`npm i -g vercel`).
- **GitHub repository** connected to a Vercel project (the project's
  Production branch should track `main`).
- **Anthropic Console account** with an API key
  (`https://console.anthropic.com/settings/keys`). v2 uses
  `claude-opus-4-7` by default.
- **Embedding provider account** — pick one:
  - **Voyage AI** (default, recommended): API key from
    `https://dash.voyageai.com/api-keys`, used with `EMBEDDER=voyage`.
  - **OpenAI** (alternative): API key from
    `https://platform.openai.com/api-keys`, used with `EMBEDDER=openai`.
- **Local toolchain**: Node 20+, npm 10+, and `psql` (only needed for the
  optional corpus sanity check below).

## First-time setup

1. **Provision Postgres.** In the Vercel dashboard go to
   *Storage → Create Database → Postgres*, accept the defaults, and link
   the new database to the Vercel project. This auto-populates
   `POSTGRES_URL` and `POSTGRES_URL_NON_POOLING` for every deployment
   environment.

2. **Provision KV (Upstash Redis).** In the Vercel dashboard go to
   *Storage → Create Database → KV*, link it to the project. This
   auto-populates `KV_REST_API_URL` and `KV_REST_API_TOKEN` (plus the
   read-only equivalents).

3. **Add application environment variables.** In *Project Settings →
   Environment Variables* add:

   | Variable | Required | Notes |
   |---|---|---|
   | `ANTHROPIC_API_KEY` | yes | Production + Preview + Development |
   | `VOYAGE_API_KEY` | yes (if `EMBEDDER=voyage`) | Production + Preview + Development |
   | `OPENAI_API_KEY` | yes (if `EMBEDDER=openai`) | Production + Preview + Development |
   | `EMBEDDER` | yes | `voyage` (default) or `openai` |
   | `NEXT_PUBLIC_SITE_URL` | yes | Set to the production URL (e.g. `https://www.michaelquintin.com`); use the Vercel preview URL for Preview scope |
   | `LLM_MODEL` | optional | Defaults to `claude-opus-4-7` |
   | `LLM_RATE_LIMIT_HOUR` | optional | Defaults from `lib/env.ts` |
   | `LLM_RATE_LIMIT_DAY` | optional | Defaults from `lib/env.ts` |

   Pick the appropriate scope (Production, Preview, Development) for each
   variable. Secrets (`*_API_KEY`) belong in all three; `NEXT_PUBLIC_SITE_URL`
   is per-environment.

4. **Pull env locally.** From the project root:

   ```bash
   vercel link
   vercel env pull .env.local
   ```

   This writes the linked Postgres, KV, and application variables into
   `.env.local` so the local dev server and CLI scripts can connect.

5. **Run database migrations.**

   ```bash
   npm run db:migrate
   ```

   Migrations are forward-only by default. Re-running is safe.

6. **Seed the corpus.**

   ```bash
   # 1. Drop initial files (.md / .pdf / .txt) into corpus/
   # 2. Edit corpus/manifest.json to register each file with id, path, tags
   # 3. Ingest:
   npm run ingest
   ```

   The ingest script chunks each file, embeds via the configured
   `EMBEDDER`, and writes rows to the `chunks` table.

7. **First deploy.** Either:

   ```bash
   vercel --prod
   ```

   or push to `main` and let the GitHub integration auto-deploy. Both
   paths run the build, the postbuild sitemap step, and the PDF resume
   build.

## Subsequent deploys

- Merge to `main` → Vercel auto-builds and deploys to Production.
- Pull requests get Preview deployments automatically; they share the
  Production Postgres and KV unless overridden in the env scope.
- The sitemap is regenerated by the `postbuild` script on every build.
- The resume PDF is regenerated by the `build-pdf` script (see 16/P08).

## Updating the corpus

1. Add or edit files under `corpus/`.
2. Update `corpus/manifest.json` with new entries (or remove stale ones).
3. Re-ingest:

   ```bash
   npm run ingest
   ```

   Pass `--rebuild` if you want to wipe and reload all chunks rather than
   incrementally updating.

4. Confirm chunks landed in Postgres:

   ```bash
   psql $POSTGRES_URL -c "select count(*) from chunks;"
   ```

5. (Optional) Spot-check voice and groundedness:

   ```bash
   npm run evals:llm
   ```

## Cost expectations

- **Anthropic API.** v2 ships with `claude-opus-4-7`, which is the
  highest-tier model. A typical chat message ranges roughly
  **\$0.02–0.10** depending on context size and answer length. The
  per-IP and per-day rate limits configured via `LLM_RATE_LIMIT_HOUR`
  and `LLM_RATE_LIMIT_DAY` keep the daily ceiling bounded — the worst
  case is roughly `LLM_RATE_LIMIT_DAY × max-cost-per-message`.
- **Voyage embeddings.** ~\$0.0001 per 1K tokens. A v2-scale corpus
  (~500K tokens) costs **<\$0.10** to ingest from scratch. Re-ingest
  cost scales linearly with token churn.
- **Vercel Postgres.** The free tier comfortably fits the v2 corpus and
  expected query volume.
- **Vercel KV.** The free tier comfortably fits v2 traffic levels and
  the rate-limit counters.
- **Total ongoing.** Bounded by the rate limits above.
  **~\$5–20/month** is the expected band at low traffic; spikes are
  capped by the per-day LLM limit.

## Rollback

- **Code/runtime rollback.** Vercel keeps every prior deployment. From
  the project's *Deployments* tab, find the last known-good deployment
  and click *Promote to Production*. No CLI or git revert required.
- **Schema rollback.** DB migrations are forward-only by default. To
  unwind a schema change, write and apply a corresponding down migration
  manually against `$POSTGRES_URL`. There is no auto-down path.
- **Corpus rollback.** Corpus is in-repo; revert the offending commit
  and re-run `npm run ingest --rebuild`.

## Smoke alerts

A read-only smoke suite (`e2e/smoke-deployed.spec.ts`) runs against the
production URL on every successful deploy via the GitHub Action at
`.github/workflows/smoke.yml`. Locally, run it against any environment with:

```bash
SMOKE_URL=https://www.michaelquintin.com npm run smoke
# or
npm run smoke -- https://www.michaelquintin.com
```

The runner prints a JSON summary on stdout and exits non-zero on any
failure. Playwright's JSON report is written to `smoke-results.json`
and uploaded as a workflow artifact.

**Hooking up alerts (out of scope for the initial implementation):**

- *Slack.* Add a `notify-slack` step to `.github/workflows/smoke.yml`
  that runs on `if: failure()` and posts to a webhook stored as the
  `SLACK_WEBHOOK_URL` repo secret. Use the
  [`slackapi/slack-github-action`](https://github.com/slackapi/slack-github-action)
  Action and include the deployment URL plus a link to the failed run.
- *Email.* Either pipe the same failure step through a transactional
  email Action (e.g. `dawidd6/action-send-mail`) using SMTP secrets,
  or rely on GitHub's per-user "failed workflow" notifications under
  *Settings → Notifications → Actions*.
- *On-call.* For pager-grade alerting, point the webhook at PagerDuty
  / Opsgenie instead of Slack and keep the Action step otherwise
  identical.

Whichever channel is wired up, it should fire on `if: failure()` only,
and include `${{ github.event.deployment_status.target_url }}` plus a
link to the Action run so the responder can jump straight to the
failing test trace.

## Troubleshooting

- **"Unauthorized" on chat** → `ANTHROPIC_API_KEY` is missing or wrong
  for the deployment's environment scope. Re-check Vercel env vars and
  redeploy.
- **"No relevant material" on every query** → corpus is empty or
  ingestion silently failed. Re-run `npm run ingest --rebuild` and
  verify with the `psql` count above.
- **429s spiking** → confirm KV is connected (the rate limiter falls
  open if KV is unreachable, so spikes there usually mean it *is*
  connected and traffic is genuinely bursty). Raise
  `LLM_RATE_LIMIT_HOUR` / `LLM_RATE_LIMIT_DAY` cautiously — every
  increase raises the daily cost ceiling.
- **PDF resume missing** → the `build-pdf` script failed during deploy.
  Check the Vercel build logs for Puppeteer/Playwright errors (missing
  Chromium, sandbox flags, font loading). Re-run locally with
  `npm run build-pdf` to reproduce.
