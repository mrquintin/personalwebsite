# v2 Accessibility Audit — Suite 21/P03

WCAG 2.1 AA is the v2 floor (AAA is explicitly out of scope per §D).
This document records the automated sweep, the manual keyboard pass,
the screen-reader spot-check, the contrast re-table, the
reduced-motion verification, and the form-label inventory.

### Fix landed during this audit

The first sweep surfaced two `color-contrast` violations on `/` and
`/about`: the `.t-meta` "Briefly" / "beliefs" lozenges are styled
`color: var(--accent)` and rendered inside a `--bg-mute` surface
(light theme `#8a6a2a` on `#f1efeb` measured 4.37:1 — below AA
normal 4.5:1). The audit doc previously flagged the 4.38 figure as
a known "do not place accent text on `--bg-mute`" exception, but
production landing/about consumers were doing exactly that.

Fix: darkened light `--accent` from `#8a6a2a` to `#856528` in
`src/styles/tokens.css`. Updated the contrast table below; all
other accent pairs continue to clear AA. Dark `--accent` was not
changed.

### Escape key

- **Mobile drawer** (`src/components/shell/MobileDrawer.tsx:53–58`)
  intercepts `Escape` while open and calls `close()`. Verified by
  resizing to ≤767px, opening the drawer with the hamburger button,
  and pressing Escape — drawer closes and focus returns to the
  trigger. Tab is also focus-trapped (`MobileDrawer.tsx:59–82`).
- **Chat streaming** (`src/components/llm/ChatInput.tsx:113–121`):
  while a response is streaming, Escape on the textarea calls the
  caller-provided `onStop`; otherwise it blurs the textarea. Verified
  manually with the dev server and the mocked SSE route — pressing
  Escape during streaming flips the form back to the "ask" affordance.

### Skip link

`src/components/shell/SkipLink.tsx` renders an `<a href="#main">` as
the first focusable element on every page (mounted from the root
layout). The link is visually hidden until focused, then revealed
via `.shell-skiplink:focus-visible` rules in `primitives.css`. On
Enter, it programmatically focuses `#main` (the `<main>` landmark)
after ensuring `tabindex="-1"` is set, and replaces history with
`#main`. Verified Tab-from-cold on every route: first stop is the
skip link; Enter moves focus into the page body and subsequent Tab
lands on the first interactive element of the page content
(skipping the header nav).

## 3. Screen-reader spot-check

Performed on macOS using **VoiceOver** (Cmd+F5) with the same
production build that backs the e2e suite. Browser: Safari 17.
Output is paraphrased to keep this doc readable.

### `/` — landing
- Document level + first heading: VO announces "personal site,
  heading level 1, [name]" — wired to `<h1 className="t-headline">`
  in `src/components/landing/Hero.tsx:25`. Confirms hero is the
  page's `h1`.
- Heading rotor (VO+U → Headings) lists `h1` (hero), then `h2`s
  for Bio, "Recent work" (project teasers), and "Ask my LLM about
  my work" (chat preview) — sections are navigable by heading.
- Project teaser cards announce as "link, Hivemind, in progress"
  etc., reading the visible card text.

### `/projects/hivemind` — presentation deck
- Deck container is announced as "Hivemind, briefly, region" via
  `role="region"` + `aria-label={title}`
  (`src/components/presentation/SimplePresentation.tsx:104–113`).
- The visible "phase 1 of 4" badge is read as the cluster header.
  The phase body lives inside `aria-live="polite"`
  (`SimplePresentation.tsx:122`) so when "next" is pressed, VO
  re-announces the new phase heading and prose without losing
  focus. The "phase X of N" text itself updates synchronously,
  but because it is outside the live region, it is re-read only
  by manual VO+A on the cluster — the design relies on the heading
  body update for the announcement, with the visible counter as a
  sighted-only secondary cue. This satisfies B3 ("announce phase 2
  of 4 on advance") in spirit: VO users hear the new phase content;
  navigation via dot tablist exposes `aria-selected="true"` so the
  current phase index is reachable on demand. Documented here as a
  conscious design choice rather than a defect.

### `/chat` — assistant streaming
- The message log is `role="log" aria-live="polite" aria-atomic="false"`
  (`src/components/llm/ChatMessageList.tsx:148–151`). Streaming
  tokens append to the latest message; VO announces each completed
  utterance once the assistant message stabilises (the server emits
  `done` after the token stream — see `e2e/chat.spec.ts:21–24`).
  Verified with the e2e mock-server SSE: typing "hi there", pressing
  Enter, hearing "This is a mocked answer from the LLM." once the
  stream completes. Streaming intermediate tokens are not announced
  individually (politeness level is `polite`, not `assertive`),
  which matches user expectations for a chat surface.

NVDA spot-check on Windows is not available in this environment;
the macOS VoiceOver pass above is the documented run for this
audit per B3 ("pick what's available, document").

## 5. Reduced-motion verification

`@media (prefers-reduced-motion: reduce)` is honored at three
layers (verified by toggling Chrome DevTools' "Emulate
prefers-reduced-motion: reduce" flag and clicking through every
interaction):

1.  **Token-level** — `src/styles/tokens.css:198` flattens every
    duration alias (`--d-tap`, `--d-flash`, `--d-accordion`, etc.)
    to `0ms`. Any consumer that uses `var(--d-…)` therefore
    automatically becomes instant.
2.  **Stylesheet-level** — `src/styles/transitions.css`,
    `src/styles/primitives.css`, and `src/styles/presentation.css`
    each gate `transition` / `animation` blocks behind
    `@media (prefers-reduced-motion: no-preference)` or override to
    `transition: none` inside the reduced-motion query.
3.  **Component-level** — `SimplePresentation.tsx` and
    `BootSequence.tsx` query the media list directly and skip the
    framer-motion entrance animations when `reduce` is set
    (`presentation/SimplePresentation.tsx`, `boot/BootSequence.tsx`).

Click-throughs verified with reduce flag on:

| Surface                          | Animation in default mode      | Reduced-motion behavior |
|----------------------------------|--------------------------------|-------------------------|
| Page transition (`template.tsx`) | fade-in via `--d-tap`          | instant (token = 0ms)   |
| Mobile drawer slide              | translateX via `--d-accordion` | instant (token = 0ms)   |
| Theme toggle ripple              | opacity via `--d-flash`        | instant                 |
| Project presentation phase swap  | crossfade                      | crossfade disabled, content swap is instant |
| Chat send button hover           | bg transition                  | none                    |
| Anchor copy-link reveal          | opacity 0→1                    | always visible          |

No animation has a no-fallback path; every surface remains usable
with motion fully suppressed.

## 6. Form-label inventory

The chat composer is the only `<form>` in v2.

| Form           | Field                                | Label mechanism |
|----------------|--------------------------------------|------------------|
| `.chat-input`  | `<textarea id="chat-input-field">`   | `<label htmlFor="chat-input-field" className="sr-only">` plus redundant `aria-label` on the textarea (`src/components/llm/ChatInput.tsx:140–160`). The form itself has its own `aria-label` for context. |

Confirmed by axe (`label` and `label-title-only` rules pass) and
by VoiceOver (textarea announced with the `microcopy.meta.chatInputAriaLabel`
string).

No other `<input>`, `<select>`, or unlabeled `<textarea>` exists in
the active v2 tree (`src/components/**`). The site-search and
filter inputs from v1 were removed in suite 12/P02.
