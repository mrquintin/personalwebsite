§B — THE CORE CONVERSION: KEYFRAME → TRANSITION

B1. Why this matters (restate for the implementer)

    CSS keyframe animations with `fill-mode: forwards` are
    a state-machine that reads: "when this rule matches,
    run this keyframe from its 0% mark to its 100% mark,
    then hold the 100% values." Crucially, when the rule
    STOPS matching (because data-state flipped) the
    animation is halted and the element reverts to whatever
    the cascade now dictates — which for .panel-body is
    the base `opacity: 0; transform: translateY(6px)`. If
    a DIFFERENT rule then matches (e.g., the panel swings
    back to expanded), a NEW animation starts, again FROM
    its 0% mark. The animation engine has no memory of the
    in-flight computed values.

    Under a rapid hover swap N→M→N (say each hop ~80ms),
    panel N's body would:

        t=0     : rule "expanded" matches; animation-in
                  starts, 260ms delay, 260ms duration.
        t=80    : rule flips to "collapsed"; animation-in
                  is yanked; animation-out starts from
                  opacity 1, transform 0 (0% of
                  animation-out). But the IN never reached
                  its visible phase (t<260ms delay), so
                  opacity was still 0 per the cascade base.
                  animation-out now says: from opacity 1 to
                  opacity 0 — a JUMP from 0 to 1 on frame 1
                  of out. Visible flash to opacity 1 for
                  one frame, then fade down.
        t=160   : rule flips back to "expanded"; animation-
                  in restarts from 0% (opacity 0, translateY
                  6px). In-flight fade-down is discarded.
                  No memory of where the fade-out got to.

    CSS transitions, by contrast, are property-level and
    from-current-value. When the target value changes, the
    transition retargets with the current computed value as
    the starting point. There is no "animation origin" to
    restart from. Rapid swap produces smooth retargeting on
    every interruption. This is the entire substance of C3.

B2. The replacement rule set (authoritative)

    Remove from globals.css (lines 119-140 of the current
    file):

      .panel-body {
        opacity: 0;
        transform: translateY(6px);
        pointer-events: none;
      }
      .panel[data-state="expanded"] .panel-body {
        pointer-events: auto;
        animation: operator-panel-in 260ms var(--ease-accordion) 260ms forwards;
      }
      .panel[data-state="collapsed"] .panel-body,

        animation: operator-panel-out 180ms linear forwards;
      }

      @keyframes operator-panel-in { … }
      @keyframes operator-panel-out { … }

    Replace with:

      /* R01+08: .panel-body is driven by CSS transitions on
         opacity and transform rather than keyframe
         animations. Transitions retarget from current
         computed value on state change, so mid-animation
         swaps do not restart the fade from keyframe origin.
         Durations preserved from R01 (260ms delay + 260ms
         fade-in; 180ms linear fade-out). */
      .panel-body {
        opacity: 0;
        transform: translateY(6px);
        pointer-events: none;
        transition:
          opacity  var(--d-panel-body-out) linear 0ms,
          transform var(--d-panel-body-out) linear 0ms;
      }
      .panel[data-state="expanded"] .panel-body {
        opacity: 1;
        transform: translateY(0);
        pointer-events: auto;
        transition:
          opacity   var(--d-panel-body-in)  var(--ease-panel-body-in) var(--d-panel-body-in-delay),
          transform var(--d-panel-body-in)  var(--ease-panel-body-in) var(--d-panel-body-in-delay);
      }
      /* collapsed and neutral inherit the base rule's
         transition (180ms linear 0ms) because neither rule
         redeclares opacity/transform targets — they fall
         back to the base's opacity:0, transform:6px. The
         transition on the base rule IS the fade-out. */

      /* keyframe definitions operator-panel-in /
         operator-panel-out are deleted — no other rule
         references them. Grep to verify before removing. */

    The @keyframes operator-fade-up and operator-blink
    blocks stay; those are used elsewhere (.cursor-blink,
    .palette-backdrop). Only operator-panel-in and
    operator-panel-out are removed.

B3. Why the transition goes on BOTH rules, not just the base

    A common temptation is to put `transition:` only on the
    base rule and let the cascade inherit it. Resist. The
    transition on the base rule (180ms linear) governs
    fade-OUT timing, which must be faster than fade-IN so
    R01's "body fades out BEFORE the width change completes"
    invariant holds (R01 §7.2). The transition on the
    expanded rule (260ms with 260ms delay) governs fade-IN
    timing, which must DELAY until the width has progressed
    ~62% so the content doesn't appear while the panel is
    still physically narrow.

    Defining the transition on both rules makes the timing
    directional: which transition applies is determined by
    which rule currently matches, which is determined by
    data-state. When data-state flips from expanded to
    collapsed, the cascade re-applies the base rule, which
    brings its transition (180ms linear) into effect for
    the NEW target values (opacity:0, translateY:6px). The
    browser sees a computed-style change from, say,
    opacity:0.4 to opacity:0, and transitions over 180ms.
    Critically, no animation restart occurs.

B4. Interruption correctness (paper proof)

    Case I — cold open, panel M: state transitions
    neutral → expanded at t=0.
      - New rule applies: opacity target 1, transform
        target (0). Transition: 260ms eased with 260ms
        delay.
      - t=0..260   : browser holds opacity at 0 (delay
                     phase), transform at 6px.
      - t=260..520 : browser eases opacity 0→1 and
                     transform 6px→0 over 260ms.
      Matches R01 §7 exactly.

    Case II — warm swap N→M: N flips expanded→collapsed at
    t=0, M flips collapsed→expanded at t=0.
      - N: base rule re-applies. Current opacity 1
        (panel was fully faded in). Transition: 180ms
        linear. Opacity 1→0 over 180ms.
      - M: expanded rule applies. Current opacity 0.
        Transition: 260ms eased with 260ms delay.
        Opacity stays 0 for 260ms (delay), then 0→1
        over 260ms.
      - Observed: N fades out over 180ms; M is
        invisible until t=260 then fades in. During
        t=0..180, only N is fading (M is still 0).
        During t=180..260, both are at their terminal
        invisible values. During t=260..520, M is
        fading in and N is fully out.
      - No flash, no flicker, no gap of neutral.

    Case III — rapid swap N→M→N within 80ms total:
      - t=0:   N flips expanded→collapsed.
               N's opacity transitions from 1 toward 0,
               180ms linear. At t=80 it's at ~0.56.
               M flips collapsed→expanded. M's opacity
               transition is in its 260ms delay phase;
               opacity stays 0.
      - t=80:  N flips BACK to expanded. The expanded
               rule re-applies. Transition: 260ms eased
               with 260ms delay. Browser sees current
               opacity 0.56 (not 0, not 1). Target: 1.
               Delay 260ms, then ease 0.56→1 over 260ms.
               M flips back to collapsed. Base rule
               re-applies. Transition: 180ms linear.
               Current 0, target 0. No-op transition.
      - t=340: N's delay ends. Opacity begins transitioning
               0.56→1 over 260ms.
      - t=600: N is fully visible.
      - Net: no flicker, no restart from 0, no spike to
        1. Total latency from N's original show to N's
        recover is ~340ms delay + 260ms eased = 600ms,
        but the user sees a smooth monotonic path from
        0.56 back to 1. This is acceptable (much better
        than the keyframe case which would have jumped
        to opacity 1 at t=80 then fallen to 0).

    Case III illustrates the delay's one remaining cost:
    under a rapid flicker back to a panel, the body
    reappearance is deferred by the delay. This is
    deliberate — R01 prescribed the delay so content
    doesn't render into a narrow frame. Under warm-swap
    N→M→N both panels' flex-basis transitions are also
    in-flight; the width is roughly correct throughout,
    and the delay is honest about "wait for width."

B5. Anti-pattern to reject

    Do NOT use `transition: all var(--d-something)`. `all`
    transitions every property that changes, including
    properties the browser internally mutates (content-box
    width under flex-basis, for instance). This can produce
    subtle double-animation artifacts. Name the properties
    explicitly: opacity and transform.

    Do NOT use CSS @property with transitionable custom
    properties as a hack to get interpolation. Transitions
    on opacity and transform are the canonical mechanism
    and compositor-friendly.

    Do NOT animate `visibility` or `display`. Neither is
    animatable across their semantic range; `display:none`
    cannot transition to `display:block`. Use opacity +
    pointer-events as we already do.

§C — PANEL.TSX CHANGES

C1. is-animating class scoping (fix for C9)

    CURRENT (Panel.tsx lines 25-31):

      useEffect(() => {
        const el = ref.current;
        if (!el) return;
        el.classList.add("is-animating");
        const t = setTimeout(() => el.classList.remove("is-animating"), 540);
        return () => { clearTimeout(t); el.classList.remove("is-animating"); };
      }, [state]);

    This adds is-animating on every state change and
    holds it for 540ms. Under a 5-tab rapid swap this
    fires five times, each with its own timer. The
    compositor ends up with will-change:flex-basis held
    on multiple panels for up to 540ms after the last
    swap settles. Five concurrent will-change hints
    cost compositor memory and defeat the point of
    scoping.

    TARGET: is-animating applies only when this panel's
    flex-basis is ACTUALLY in flight — i.e., when state
    just changed AND the CSS transition is running. On
    every state change is-animating is (re-)added with a
    fresh timer; the timer's duration must match the
    longest transition the panel can run. That duration
    is `--d-accordion` (420ms). The current 540ms was a
    hand-tuned overestimate; tighten it to 420ms and
    drop the coupling to the old keyframe budget.

    REPLACEMENT:

      useEffect(() => {
        const el = ref.current;
        if (!el) return;
        el.classList.add("is-animating");
        // Match --d-accordion. Read from CSS if you want to
        // stay token-driven:
        //   const cs = getComputedStyle(document.documentElement);
        //   const ms = parseInt(cs.getPropertyValue("--d-accordion")) || 420;
        // For simplicity and test determinism, hard-code 420
        // here and pair with the tokens.css value.
        const t = setTimeout(() => el.classList.remove("is-animating"), 420);
        return () => { clearTimeout(t); el.classList.remove("is-animating"); };
      }, [state]);

    This is technically the same pattern (per-panel timer
    on every state change) but shorter. The deeper
    correctness claim — "only the panels actually
    animating have is-animating" — is already guaranteed
    because only panels whose `state` changed get a state
    prop change; React won't re-run the effect on
    unchanged props.

    OPTIONAL tighter scoping (implement if straightforward):
    use a transitionend listener to remove is-animating
    when the browser actually completes (or cancels) the
    flex-basis transition:

      useEffect(() => {
        const el = ref.current;
        if (!el) return;
        el.classList.add("is-animating");

        let fallbackTimer: number | null = null;
        const onTransitionEnd = (e: TransitionEvent) => {
          if (e.target !== el) return;            // ignore
                                                  // descendant
                                                  // transitions
          if (e.propertyName !== "flex-basis") return;
          el.classList.remove("is-animating");
        };
        el.addEventListener("transitionend", onTransitionEnd);

        // Safety net: if transitionend never fires (rare,
        // but possible under tab visibility changes or
        // interrupted transitions that don't emit the event
        // in some engines), remove the class after
        // --d-accordion + a safety margin.
        fallbackTimer = window.setTimeout(
          () => el.classList.remove("is-animating"),
          420 + 80
        );

        return () => {
          el.removeEventListener("transitionend", onTransitionEnd);
          if (fallbackTimer) clearTimeout(fallbackTimer);
          el.classList.remove("is-animating");
        };
      }, [state]);

    The transitionend-driven variant is preferred
    because it guarantees will-change drops the moment
    the compositor is done with the flex-basis
    transition, reducing the window where unused
    will-change hints cost memory. Adopt this variant
    unless test flakes appear on Safari (Safari has
    historically been less reliable about emitting
    transitionend on cancelled transitions; if so, fall
    back to the simpler timer).
