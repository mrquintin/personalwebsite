P02 — MODE 1, THREE ORDERS SORTER
====================================

ROLE: You are implementing MODE 1 of the Theseus Deep experience.
The Three Orders Sorter is the orientation mode: by the time a
user has advanced through the card deck, the first/second/third-
order distinction is understood, and Theseus's own position at
the boundary of the second and third order is legible.

Read P01_chassis_and_panel_host.txt first. This
prompt assumes the TheseusExperience shell, ModeTabBar, and the
zustand store shape are already in place. This prompt fills in
OrdersSlice, authors the card deck, and implements the sorter
components.

---

GROUNDING

All content in this mode is grounded in:

  TH1 §2   — the three orders, precisely
  TH1 §3   — what the firm actually does (faculties reference)
  TH2 §1   — the problem the meta-method solves (framing)
  TH6 §10  — what the seven projects are not ("each project is
             methodological. None of them makes a substantive
             claim about the world")
  TH6 §12  — the point of all of this (firm's own position)

The canonical definitions (verbatim from TH1 §2):

  "First order — Substantive claims. 'We believe X is true.'
  The product is a claim about the world. The vulnerability is
  the obvious one: when X is false, the firm collapses."

  "Second order — Methodology. 'We have a method for
  determining whether X is true.' The product is a procedure:
  a coherence engine, a contradiction detector, an adversarial
  generator. A false X does not collapse the firm; the method
  either catches the falsity or is shown to be miscalibrated."

  "Third order — Meta-methodology. 'We have a theory of what
  makes a method reliable, and we can evaluate competing
  methods by that theory.' The product is not a claim and not
  a procedure but a criterion for evaluating procedures."

Theseus's own position (verbatim from TH1 §2):

  "Theseus operates at the boundary between the second and
  third orders. The existing engineering (Noosphere, Dialectic,
  the Codex) is a second-order product: it operationalizes
  specific methods. The active intellectual work (the seven
  projects described in Guide VI) is third-order: it produces
  methods for evaluating methods."

All three of these passages MUST land verbatim in
src/lib/theseus/sources.ts with the keys:
    TH1_SEC_2_ORDER_1
    TH1_SEC_2_ORDER_2
    TH1_SEC_2_ORDER_3
    TH1_SEC_2_POSITION

The mode renders them in its reveal panels and its capstone
card. Non-verbatim paraphrase in user-facing prose is permitted
elsewhere but these four passages must be exact.

C) THE CARD DECK (AUTHORED — src/lib/theseus/ordersDeck.ts)

C1. Types

    export type Order = "first" | "second" | "third";

    export interface DeckCard {
      id: string;
      kind: "sort" | "capstone";
      statement: string;           // shown on the front of the card
      canonicalOrder: Order | null;  // null for the capstone
      rationale: string;           // shown after reveal
      consequenceIfMisclassified: string;  // one-sentence note
      sourceAnchor: string;        // e.g., "TH1 §2; TH6 §10"
    }

    export const DECK: DeckCard[] = [ /* 11 sort cards + 1 capstone = 12 */ ];

C2. Authoring rules for sort cards (IDs deck-01 through deck-11):

    - Each statement is 1-2 short sentences. No statement
      invents policy or technology that is not in TH1-TH6.
    - Each canonicalOrder is determinate and defensible; no
      "ambiguous" cards in the main deck. (Intentional
      ambiguity is reserved for the capstone.)
    - rationale is 2-3 sentences grounded in the source. If it
      paraphrases, it names the source section.
    - consequenceIfMisclassified is one sentence that makes the
      practical cost of the misclassification vivid. For
      example, classifying a first-order claim as methodology
      yields "the firm would defend a specific prediction as
      if defending its method; a false prediction would then
      wrongly be read as a method failure".
    - sourceAnchor names 1-2 TH<n> §<x> locations.

D2. OrderCard.tsx (sort card lifecycle)

    Three-phase render:

                           the <OrderChoice /> control is
                           active; reveal is hidden.
      Phase 2 — CHOSEN:    user has picked an order; the

D3. OrderChoice.tsx (the three-way control)

    Three side-by-side buttons: FIRST / SECOND / THIRD.
    On larger widths, each button has a short inline
    description ("substantive claim" / "methodology" /
    "meta-methodology"). On NARROW, descriptions collapse
    into popovers that open on focus.

    Keyboard: 1/2/3 select first/second/third; arrow keys
    move focus; ENTER commits.

    Visual treatment: when revealed, the button matching
    canonicalOrder gets a 2px left border in the "correct"
    color; the user's choice (if different) gets a 2px
    left border in the "mismatch" color. NO tick/cross
    emoji; NO "correct/incorrect" copy. The reveal is
    about the rationale, not the score. The mode does NOT
    track a running score.

D4. OrderReveal.tsx

    Renders in order:
      1. CANONICAL ORDER badge — "FIRST-ORDER", "SECOND-

         TH1_SEC_2_ORDER_1 / ORDER_2 / ORDER_3). Rendered

D5. OrdersCapstone.tsx

           - To the Refusal Panel (mode 4 — method under

    (iii) NavControls on the capstone show a RESTART button

D6. DeckProgress.tsx

H) TYPE GUARDS

    In src/lib/theseus/typeGuards.ts:

    export function typeGuardForCardDeck(x: unknown): DeckCard[] {
      if (!Array.isArray(x)) throw new Error("DECK not array");
      for (const c of x) {
        if (typeof c.id !== "string") throw new Error(`card.id`);
        if (c.kind !== "sort" && c.kind !== "capstone") {
          throw new Error(`card.kind ${c.id}`);
        }
        if (typeof c.statement !== "string") throw new Error(`card.statement ${c.id}`);
        if (c.kind === "sort" && !["first","second","third"].includes(c.canonicalOrder as string)) {
          throw new Error(`canonicalOrder ${c.id}`);
        }
        if (c.kind === "capstone" && c.canonicalOrder !== null) {
          throw new Error(`capstone canonicalOrder must be null`);
        }
        for (const k of ["rationale","consequenceIfMisclassified","sourceAnchor"]) {
          if (typeof (c as any)[k] !== "string") throw new Error(`card.${k} ${c.id}`);
        }
      }
      if (x.length !== 12) throw new Error(`DECK must have 12 entries`);
      if (x[11].kind !== "capstone") throw new Error(`deck-12 must be capstone`);
      return x as DeckCard[];
    }

    Invoke from ordersDeck.ts at module load:

      import { typeGuardForCardDeck } from "./typeGuards";
      const _raw = [ /* the 12 cards */ ];
      export const DECK = typeGuardForCardDeck(_raw);
