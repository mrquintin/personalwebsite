M06 — HVM PANEL (narrow)
================================================

PURPOSE
-------
Narrow-mode rendering for the Hivemind panel (02,
/hivemind). HVM has TWO faces sharing one ExperienceHost:
the strategic face (05 suite — deliberation theater, knob
bench, four-failures map, knowledge pipeline, audit
artifact) and the technical face (06 suite — Coherence
Engine: layer scoring lab, cosine paradox, domain-relative
comparator, decision-policy gates with the §C8 regime
selector, copy + fixture integrity).

HVM is the most information-dense panel and the hardest to
narrow well. This prompt is correspondingly the longest
per-panel prompt.

§A  FACE TOGGLE ON NARROW
=========================

Desktop behavior: the HVM panel, when expanded, renders a
face-toggle (strategic / technical) near the top; the two
faces share state via the ExperienceHost.

Narrow behavior: preserve the face toggle. It becomes a
two-button segmented control anchored at the top of the
panel body, 100% width, 44 px tall, monospace labels
"STRATEGIC" and "TECHNICAL" uppercase. Active state: --accent
background, --bg-0 text. Inactive: --bg-2 background,
--fg-mute text.

  Render:
    const { mode } = useExperienceHost();
    const [face, setFace] = useState<"strategic" | "technical">("strategic");
    return (
      <>
        <FaceToggle face={face} onSwitch={setFace} />
        {face === "strategic"
          ? <StrategicFaceNarrow />
          : <TechnicalFaceNarrow />}
      </>
    );

  Announce on face switch: `${face} face selected`.

The face toggle is NOT a tab. It is a segmented control; tab
semantics (arrow-key navigation between tabs) are not applied.
On narrow, tap switches faces.


§B  STRATEGIC FACE (05 suite) — NARROW
======================================

B1 — DELIBERATION THEATER (05/P01)
-----------------------------------
Desktop behavior: N agents arranged in a circle or grid,
proposing / critiquing / voting; a central log scrolls with
the deliberation transcript.

Narrow behavior: one-agent-at-a-time with a top strip.

  Render:
    - TOP STRIP: a horizontal row of agent avatars (small
      circles with initials), 36 px diameter each, 8 px gap,
      touch-action: pan-x for overflow. The currently
      focused agent has a --accent border. Tap an avatar to
      focus that agent.
    - MAIN: the focused agent's CARD:
      - name + role (strategist / critic / pragmatist / etc.)
      - current stance (a 1–2 line quote pulled from the
        current deliberation state)
      - a .prp-prose block with the agent's latest contribution
    - BELOW: a transcript of the deliberation, scrolling
      from oldest to newest. Each line is "AGENT NAME :: text";
      the focused agent's lines are colored --accent.

  Animation: the current desktop choreography has staggered
  entries per agent (05/P01 spec). On narrow, reducedMotion
  is default true (M03 §E). Under reducedMotion, deliberation
  steps happen instantly. Under explicit opt-in motion, the
  transcript fades new lines in at 180 ms.

  No circular/arranged-agent visual on narrow. The
  one-at-a-time agent card + transcript is the mobile
  deliberation UX.

  ACCEPTANCE:
    [ ] DeliberationTheaterNarrow.tsx exists.
    [ ] Tap on an avatar focuses that agent and updates the
        main card within 200 ms.
    [ ] Transcript is the page-scroll, not a nested scroller.

B2 — KNOB BENCH (05/P02)
------------------------
Desktop behavior: a bench of parameter knobs that the user
twists to change deliberation behavior (temperature,
consensus threshold, adversarial weight, etc.).

Narrow behavior: a vertical stack of labeled range sliders.

  Each knob becomes a labeled row:
    - label (mono, uppercase, --t-xs-size, --fg-mute)
    - numeric value (mono, right-aligned, --t-sm-size, --fg)
    - slider (native <input type="range">, full width, min
      and max labels below left/right)

  Why native range slider, not a custom knob control:
    - Native sliders have excellent touch behavior on both
      iOS Safari and Chrome Android.
    - They respect user accessibility preferences.
    - They are keyboard-operable without custom code.
    - A custom circular knob at <44 px is unusable on touch.

  Knob bench on narrow has AT MOST 6 knobs visible. If the
  desktop version has more, group the extras into an
  "Advanced" accordion that starts collapsed.

  ACCEPTANCE:
    [ ] KnobBenchNarrow.tsx exists.
    [ ] Each knob is a native range slider with ≥44 px touch
        height (use `touch-action: pan-x` and
        `height: 44px` with a `::-webkit-slider-thumb`
        size of 24 px).
    [ ] Changing a knob announces
        `${label} set to ${value}` via the host's announce().

B3 — FOUR FAILURES MAP (05/P03)
-------------------------------
Desktop behavior: a 2×2 grid of failure modes (groupthink,
hallucination-cascade, oscillation, false-consensus) with
mini-vignettes per cell and connecting lines.

Narrow behavior: a stack of four failure cards with a small
"diagnosis" tool at the top that, given a deliberation
fixture, highlights which failure is present.

  Render:
    - TOP: a fixture picker (one-line select: "Choose a
      deliberation to diagnose"), rendering a native <select>
      styled to match the dark theme.
    - Below: a row of four mini-cards, one per failure mode,
      each 44 px tall, dimmed by default. When the picker
      changes, the card matching the diagnosed failure
      lights up with --accent border.
    - Below: the highlighted failure's full content in
      .prp-prose.

  The connecting-lines visual from desktop is dropped on
  narrow (no visual mechanism for a 2×2 adjacency graph in
  a single column).

  ACCEPTANCE:
    [ ] FourFailuresNarrow.tsx exists.
    [ ] Native <select> is used for the fixture picker.
    [ ] Diagnosed failure's card lights up within 200 ms of

B5 — AUDIT ARTIFACT (05/P05)
----------------------------
Desktop behavior: a generated audit document rendered in a
typewriter style with annotations in the margin.

Narrow behavior: reading-mode typography; annotations as
inline footnote numbers (not margin notes).

  Render:
    - The document text in .reading-mode or .prp-prose.
    - Annotations replaced with ¹ ² ³ superscript markers,
      inline with the text they annotate.
    - At the bottom of the document, a footnotes block
      listing each numbered annotation.
    - Tap a superscript → scrolls to the footnote; tap the
      footnote's "↵ back" affordance → scrolls back to the
      marker.

  The margin-notes pattern fails on narrow because there is
  no margin. Footnotes preserve the annotation content.

  ACCEPTANCE:
    [ ] AuditArtifactNarrow.tsx exists.
    [ ] No margin notes on narrow; all annotations are
        inline-numbered footnotes.

C3 — DOMAIN-RELATIVE COMPARATOR (06/P03)

C4 — GATE CHAIN (06/P04) — THE SIX GATES
----------------------------------------
Desktop behavior: six gates (quality, compliance, anti_gaming,
portfolio, coherence, confidence) rendered horizontally in
fixed order with pass/fail badges and inline micro-explanations.

Narrow behavior: a vertical accordion-within-accordion. Each
gate is a 56 px row with:
  - gate index (1–6) in a 24 px circle at the left
  - gate name (mono uppercase --t-xs-size)
  - pass/fail badge on the right (a 12 px square, --ok
    or --danger, with an ✓ or ✗ glyph)
  - tap → expands the row in place to reveal:
    - the gate's check predicate (one line)
    - the current fixture's input values for that predicate
    - a one-line explanation of why this run passed or failed

  The per-gate expand transition mirrors the outer accordion's
  max-height animation but is faster (200 ms) because the
  content is smaller.

  Only ONE gate can be expanded at a time within the chain.
  Tapping a second gate collapses the first.

  Fixed order is enforced in rendering. Do NOT permit
  rearrangement on narrow.

  ACCEPTANCE:
    [ ] GateChainNarrow.tsx exists.
    [ ] Six rows in fixed order.
    [ ] Pass/fail badges visible at the row level without
        expanding.
    [ ] Expanding a gate reveals its predicate + inputs; only

§D  PERFORMANCE
===============

HVM is heavy. The 05 + 06 suites together hit the 80 KB gzip
per-experience budget (03/I01 LAZY-LOAD CONTRACT). Narrow
variants must NOT increase this budget. Strategies:

  - Narrow components import the SAME fixture modules as
    desktop (no duplication).
  - Narrow components reuse primitives from the 05/06
    suites where possible (e.g., formula rendering, SVG
    helpers) — they just compose them differently.
  - Carousel / swipe libraries: do NOT add a third-party
    carousel dependency. Use CSS scroll-snap + a tiny
    useScrollPosition hook. Swiper.js and equivalents add
    ~20 KB gzip; we can't afford it.
  - Chart rendering on C3 (domain comparator) uses a custom
    ~400-line SVG renderer, NOT Chart.js or D3. Drawing a
    log-axis with 2 series is O(100 LOC).

Budget: the addition of all narrow variants together should
add ≤8 KB gzip to the HVM experience bundle. M09 validates.


§E  STATE CONTINUITY
====================

A user who expands the HVM panel on narrow and selects the
technical face, adjusts a knob, and sets regime=defensive,
then rotates to landscape (mode: compact) — the state
persists. This is ensured by:

  - Per 06/P04 §A2, the Coherence Engine store (zustand)
    holds activeRegime and related state. The store is
    not remounted on resize. Narrow and non-narrow renders
    read the same store.
  - Strategic-face state (selected agent, knob values,
    diagnosis fixture) lives in the 05 suite's equivalent
    store. Same persistence rule.
  - The face toggle (§A) state does NOT persist across
    remounts unless it is lifted to the 05/06 combined
    store or a sessionStorage key. RECOMMENDATION: use
    sessionStorage["hvm.face"] = "strategic" | "technical".
    Inexpensive, survives soft navigations.


§F  ROUTE /hivemind ON NARROW
=============================

The route page renders the same face toggle + same narrow
components, full-bleed. The ExperienceHost surface is "route"
but the component tree is identical. The regime selector's
sticky-bottom positioning works on both surfaces (accordion
panel body and route page both have a scrolling parent = the
page).
