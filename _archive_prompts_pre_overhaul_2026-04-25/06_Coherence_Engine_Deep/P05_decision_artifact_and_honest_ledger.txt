P05 — DECISION ARTIFACT & HONEST LEDGER
=========================================

ROLE: Continuing as the senior TypeScript/React engineer on the
Coherence Engine suite. This prompt implements MODE 5, the
Decision Artifact & Honest Ledger — a mode that has two jobs:

  (i)  Show the user, as a working thing, the reproducible
       decision artifact that CE4 §5 describes. A REPLAY button
       re-derives the artifact hash in front of the user,
       producing the byte-identical SHA-256 (or truncation
       thereof) stored with the fixture.

  (ii) Render, with equal visual weight, the three candid limits
       from CE1 §8 and CE5 §10.4. The thesis of this mode is that
       the honest caveats DESERVE the same rigor of presentation
       as the engineering claim. Without (ii), (i) becomes
       marketing.

Read P01, P02, P03, and P04 before starting. This
mode depends on the fixtures from P04 (decisionInputs.ts and
their gate-run results).


---

WHAT THIS MODE HAS TO COMMUNICATE

  - Every decision produces a JSON artifact with a deterministic
    hash (CE4 §5). Replay of the same input against the same
    policy version yields a BIT-EXACT identical artifact.

  - The artifact contains the full DecisionInput, the policy
    version, parameter set id, CSreq with its full decomposition
    (domain base + log-scale term + seven R components),
    CSobserved, the outcome, the reason codes, and (on pass) an
    escalation-packet reference.

  - scoring_mode is either "enforce" (decision has effect) or
    "shadow" (computed and recorded but no downstream action).
    Shadow lets the team deploy new parameter sets against live
    traffic without touching funding decisions (CE4 §7).

  - The three candid limits from CE1 §8 remain true:
      (1) Predictive validity is unproven.
      (2) The layer weights are partly principled and partly
          tuned.
      (3) Cross-domain contradiction detection degrades.

  - The Quintin Hypothesis is an empirical hypothesis, not a
    measured fact (CE1 §3.2, CE5 §10.4).

  - A planned 500-startup study (250 succeeded, 250 failed,
    p < 0.01) is the stated empirical-validation program, not
    yet reported complete (CE1 §3.2).

    Each artifact is accompanied by a small precomputed canonical
    serialization and its hash. The canonical form is produced by
    a deterministic, field-ordered JSON serializer (NOT
    JSON.stringify, which is order-dependent across runtimes).

B2. Canonical serializer

    src/lib/coherenceEngine/hashStub.ts exports:

       export function canonicalJson(value: unknown): string;
       export async function sha256Hex(input: string): Promise<string>;

    Implementation notes:

      - canonicalJson performs deterministic key ordering
        (lexicographic), expands numbers without trailing zeros,
        handles null, booleans, strings, integers, floats, arrays,
        objects. No Date, Map, Set, or class instances; they are
        rejected with a thrown error during tests so fixture bugs
        fail loud.

      - sha256Hex uses the Web Crypto API
        (crypto.subtle.digest("SHA-256", ...)). No third-party
        hash library in the bundle. On browsers without
        crypto.subtle (none of the target browsers lack it) the
        REPLAY button is disabled with an explanatory note; this
        fallback is a safety measure, not an expected path.

    For each fixture, the committed artifact_hash is the
    sha256Hex of canonicalJson of the artifact with
    artifact_hash itself removed. The hashStub helper handles
    this by omitting the field before hashing.

    P06's ship-gate includes a step that runs sha256Hex over
    every committed fixture artifact and asserts equality with
    the stored artifact_hash. Drift fails the build.

C3. ReplayButton.tsx

    A button labeled [REPLAY]. Click behavior:

      1. setReplayState("running")
      2. announce("Replaying decision for {application_id}...")
      3. build the artifact object from the fixture input and
         the precomputed gate-run-result from P04
      4. compute canonicalJson(artifact without artifact_hash)
      5. compute sha256Hex of that string
      6. compare to the stored artifact_hash
      7. if equal: setReplayState("complete"),
         lastComputedHash = <computed hex>
      8. if not equal: setReplayState("mismatch"),
         lastComputedHash = <computed hex>

    Visual feedback:
        running  → pulsing "hashing..." text, keyboard-disabled
        complete → the HashPanel highlights "expected == computed"
                   in a green-equivalent terminal color
        mismatch → HashPanel renders both hashes with a diff
                   highlight and a small "fixture drift — run P06
                   ship-gate" note. A mismatch in production would
                   be a ship-blocker; in this mode, it remains
                   displayed so the user sees what drift looks
                   like.

    The point of the user-visible replay is epistemic: rather
    than asserting "reproducibility is a property of the
    system", the mode demonstrates it in front of the user and
    does so fast (the hash completes well under 100ms for the
    artifact sizes we commit).

D) STATE TRANSITIONS & EVENTS

D1. runReplay implementation sketch

    async runReplay() {
      set({ replayState: "running" });
      const { activeArtifactId, scoringMode } = get().audit;
      const artifact = buildArtifact(activeArtifactId, scoringMode);
      const bodyJson = canonicalJson(omit(artifact, ["artifact_hash"]));
      const computed = await sha256Hex(bodyJson);
      const expected = (scoringMode === artifact._meta.originalMode)
        ? artifact.artifact_hash
        : artifact._meta.shadowHash;   // precomputed alt hash
      const ok = computed === expected;
      set({
        lastComputedHash: computed,
        replayState: ok ? "complete" : "mismatch",
      });
      announce(
        ok
          ? "Replay complete. Hash matches."
          : "Replay complete. Hash mismatch. See panel."
      );
    }

    Note: artifacts.ts must commit TWO hashes per fixture — the
    enforce-mode hash (as the canonical artifact_hash) and the
    shadow-mode hash (stored in a _meta sidecar). This way,
    toggling scoringMode still produces a verifiable match
    rather than a mismatch. The _meta field is stripped before
    canonicalization to keep it off the hash.

D2. Announcements

    replay running   — "Replaying decision for {id}..."
    replay complete  — "Replay complete. Hash matches."
    replay mismatch  — "Replay complete. Hash mismatch."
    scoring-mode flip — "Scoring mode: {enforce|shadow}."
    ledger section expand — "Ledger item expanded: {anchor}."

D3. Keyboard

    r          trigger REPLAY
    e / s      set scoring mode to enforce / shadow
    1 / 2 / 3  expand/collapse ledger item 1/2/3
    Enter/Space activate focused control
    Esc        collapse all expanded sections back to defaults
